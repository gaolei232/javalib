# 基于 Vue 的图书馆座位预约系统的研究与实现

**摘要**：随着高校图书馆座位资源日益紧张，传统的人工管理方式存在信息滞后、预约冲突频发等问题。本文基于前后端分离架构，设计并实现了一套面向考研与高频自习场景的图书馆座位预约系统。系统采用 Vue 3 构建单页面应用前端，Spring Boot 3 提供 RESTful 后端服务，Spring Data JPA 配合 MySQL 实现数据持久化。在预约控制方面，系统通过时间区间重叠判定算法实现细粒度的冲突检测，确保同一座位在同一时间段内不会被重复预约。在状态同步方面，系统基于 WebSocket 实现了座位状态的实时推送，使客户端能够及时获得座位变更通知。此外，系统还集成了诚信指数评价机制、签到验证和学生统计等功能。本文从需求分析、系统设计、详细实现和测试验证四个阶段对系统进行了全面阐述。测试结果表明，系统前端构建耗时1.12秒，后端 JAR 包体积约62MB，核心接口功能正确，冲突检测与状态广播链路完整。本文的研究为前后端分离架构在资源预约类系统中的应用提供了可参考的实践案例。

**关键词**：图书馆座位预约；前后端分离；Vue 3；Spring Boot；WebSocket；冲突检测

## 1 绪论

### 1.1 研究背景

近年来，随着高等教育规模的持续扩大，高校图书馆的座位资源面临着日益严峻的供需矛盾。考研复习、公务员备考、专业资格考试等各类学习任务使得图书馆自习区域的使用需求大幅增长，尤其是考研自习区域，座位紧张问题尤为突出。每到考研冲刺阶段，"占座难"成为困扰学生和图书馆管理方的共同难题。

传统的图书馆座位管理主要依赖人工方式进行。学生通过自行寻找座位的方式使用自习空间，管理员难以实时掌握各座位的使用状态。这种方式存在若干突出问题：第一，信息滞后，学生需要到现场才能知道是否有空位，时间成本较高；第二，预约冲突，缺乏统一的预约协调机制，同一座位可能被多人同时占用；第三，管理成本高，管理员需要定期巡查和清理占座行为，工作效率低下。

随着 Web 前端技术的发展，Vue 3 等现代前端框架为构建交互丰富的单页面应用提供了良好的技术基础。同时，Spring Boot 等后端框架的成熟使得 RESTful 服务的开发效率大幅提升。将前后端分离架构引入座位预约场景，构建一套信息透明、控制精准、实时同步的预约系统，对于缓解高校图书馆座位管理难题具有现实意义。

### 1.2 国内外研究现状

国内关于图书馆座位预约与学习空间管理的研究，主要集中在预约规则设计、平台接入方式和服务流程改造等方面。沈梦轩、游晓丹围绕公共图书馆预约占座系统的构建，对基于微信公众号的预约服务模式进行了设计分析，认为预约入口向移动端转移是提升用户体验的有效方向。部分高校图书馆采用第三方平台（如"我去图书馆"等小程序）实现座位的预约与管理，这类平台能够完成基本的预约流程，但在与校内身份认证系统的对接、预约规则的灵活配置等方面存在限制。

在技术方案层面，现有的图书馆座位预约系统可大致分为三类。第一类是基于单体架构的传统 Web 系统，前后端代码耦合较紧，页面交互以服务端渲染为主。第二类是采用前后端分离架构的系统，前端负责界面展示和交互逻辑，后端专注业务数据处理，二者通过 RESTful 接口通信。第三类是移动端原生或混合应用，以小程序和 App 为主要形态，通常依赖第三方平台的接口能力。

国外对于共享学习空间的研究起步较早，部分高校图书馆已建立较为成熟的座位和研讨室预订系统。例如，美国部分高校采用在线预约平台，允许学生在规定时间内预订图书馆的研讨室和自习座位，并设置了签到、违约计分等管理规则。这些系统在预约规则的设计上较为精细，但通常基于特定的商业或开源平台开发，与国内高校管理模式的适配性有限。

综合国内外研究与实践可见，图书馆座位预约系统已经具备较为清晰的业务基础，但仍存在若干值得进一步研究的问题。第一，不少研究对预约逻辑的描述较为宏观，缺少面向具体实现的冲突控制策略分析。第二，部分系统能够完成预约记录的存储，却难以及时反映实时座位状态的变化，缺乏有效的状态同步机制。第三，在学生管理和服务优化方面，诚信评价和数据统计分析的支持还不够深入。本文围绕以上问题展开设计与实现工作。

### 1.3 研究内容与目标

围绕图书馆座位预约业务，本文的研究内容主要包括以下几个方面。

第一，对考研与高频自习场景下的图书馆座位使用需求进行分析，明确普通用户和管理员两类角色的核心业务需求，界定系统应覆盖的功能边界。

第三，在具体实现层面对用户认证、座位预约、冲突检测和实时状态同步等关键模块进行实现分析，展示核心代码的设计思路。

第四，通过实际的构建验证和接口测试对系统功能进行检验，形成完整的测试证据链。

## 2 相关技术与理论基础

### 2.1 Vue 3 前端框架

Vue 3 是由尤雨溪及其团队开发的前端 JavaScript 框架，于2020年发布正式版本。相比于 Vue 2，Vue 3 在响应式系统、组合式 API 和 TypeScript 支持等方面进行了重大改进。

Vue 3 的核心特性包括以下几个方面。第一，基于 Proxy 的响应式系统。Vue 2 使用 Object.defineProperty 实现数据响应式，无法检测属性的新增和删除。Vue 3 改用 Proxy 代理对象，能够拦截对象的任意属性操作，响应式行为更加全面和高效。第二，组合式 API（Composition API）。Vue 2 使用选项式 API，组件的逻辑按 data、methods、computed 等选项进行组织，当组件逻辑较为复杂时，同一功能的代码可能分散在不同的选项中。Vue 3 的组合式 API 允许按功能组织代码，通过 setup 函数和 ref、reactive 等响应式 API 实现逻辑的灵活组合和复用。第三，编译优化。Vue 3 引入了静态标记（PatchFlag）和块树（Block Tree）等编译优化策略，在虚拟 DOM 的 diff 过程中能够跳过静态内容的比较，提升了渲染性能。

在状态管理方面，Vue 3 官方推荐使用 Pinia 作为状态管理库。Pinia 提供了类型安全、模块化的状态管理方案，每个 store 独立定义，避免了单一全局 store 膨胀的问题。在本项目中，Pinia 用于管理用户认证状态、应用配置等全局数据。

### 2.2 Spring Boot 后端开发框架

Spring Boot 是 Pivotal 团队在 Spring 框架基础上推出的快速开发框架，旨在简化 Spring 应用的初始搭建和配置过程。Spring Boot 遵循"约定优于配置"的设计理念，通过自动配置机制大幅减少了手动配置的工作量。

Spring Boot 的核心优势体现在以下方面。第一，起步依赖（Starter）。Spring Boot 将常用的依赖组合封装为起步依赖，例如 spring-boot-starter-web 包含了 Spring MVC 和嵌入式 Tomcat，spring-boot-starter-data-jpa 包含了 Hibernate 和 Spring Data JPA。开发者只需引入一个起步依赖即可获得一组协调兼容的依赖版本。第二，自动配置。Spring Boot 根据 classpath 中的依赖和环境配置自动创建和配置 Bean，开发者可以在 application.properties 或 application.yml 中通过属性覆盖默认配置。第三，嵌入式服务器。Spring Boot 内嵌了 Tomcat、Jetty 等 Servlet 容器，应用可以直接以 JAR 包形式运行，无需额外部署 WAR 文件。

本系统中，Spring Boot 3.2.0 作为后端基础框架，集成了 Spring MVC 处理 HTTP 请求、Spring Data JPA 管理数据持久化、Spring Security 提供安全支持，并利用 spring-boot-starter-websocket 实现 WebSocket 通信。

### 2.3 Spring Data JPA 与 MySQL

Spring Data JPA 是 Spring Data 家族中用于简化 JPA（Java Persistence API）数据访问的模块。它通过在接口中定义方法签名来自动生成查询逻辑，开发者无需编写具体的实现代码。

Spring Data JPA 的核心机制包括：第一，实体映射。通过 @Entity、@Table、@Column 等注解将 Java 对象与数据库表建立映射关系，开发者可以面向对象的方式操作数据。第二，派生查询。根据方法命名规则自动生成查询语句，例如 findBySeatId(String seatId) 会自动生成按 seatId 字段查询的 JPQL 语句。第三，@Query 注解支持自定义 SQL 或 JPQL 查询，用于处理复杂的数据检索场景。

MySQL 是目前使用最广泛的开源关系型数据库之一，具有稳定可靠、社区活跃、性能优良等特点。本系统中，MySQL 作为主要数据库存储用户、座位、预约记录等业务数据，同时在开发环境中使用 H2 内存数据库便于测试。

### 2.4 WebSocket 实时通信

WebSocket 是一种在单个 TCP 连接上实现全双工通信的协议，由 IETF 在 RFC 6455 中标准化。与传统的 HTTP 请求-响应模式不同，WebSocket 允许服务器主动向客户端推送消息，是实现实时数据同步的关键技术。

WebSocket 的通信过程分为握手和数据传输两个阶段。握手阶段通过 HTTP Upgrade 机制将协议从 HTTP 切换为 WebSocket。在数据传输阶段，客户端和服务器之间以帧为最小单位进行双向通信，帧类型包括文本帧、二进制帧、ping/pong 心跳帧和关闭帧。

在 Spring Boot 中实现 WebSocket 服务端需要以下步骤：实现 WebSocketHandler 接口或继承 TextWebSocketHandler 类，重写 afterConnectionEstablished、handleTextMessage 和 afterConnectionClosed 等方法；通过 WebSocketConfig 注册 handler 并配置端点路径；利用 ConcurrentHashMap 维护会话集合以便进行消息广播。

### 2.5 前后端分离与状态管理

前后端分离是一种软件架构模式，核心思想是将前端展示层与后端业务层解耦，二者通过 HTTP 接口进行数据通信。在这种模式下，前端负责用户界面渲染和交互逻辑，后端专注业务规则和数据处理。

前后端分离架构的优势在于：第一，职责清晰，前后端团队可以并行开发，互不干扰；第二，复用性强，后端接口可以同时服务于 Web 端、移动端等多种前端；第三，部署灵活，前后端可以独立部署和扩展。

在前端状态管理方面，本系统采用 Vue Router 实现前端路由，通过路由守卫控制页面访问权限。Pinia 认证 store 管理用户令牌和角色信息，利用 Axios 拦截器自动在请求头中添加 Authorization 字段，实现身份认证的全局处理。

## 3 系统需求分析

### 3.1 系统应用场景分析

本系统的目标场景是高校考研自习室和图书馆高频自习区域的座位管理。此类场景具有以下几个特征。

使用时段集中。考研自习高峰期，学生通常会在图书馆开馆前即开始排队，座位在开馆后短时间内被迅速占满。传统的"先到先得"模式导致学生需要花费大量时间排队等待，且无法预知座位情况。

预约时段灵活。学生的自习需求并非固定为"全天"，部分学生可能仅在上午、下午或晚间有自习计划。灵活的时段预约机制能够提高座位的周转效率，允许不同学生在不同时段使用同一座位。

管理规则复杂。考研自习区域通常需要设置一定的使用规则，例如限定最长预约时长、要求签到确认、对违约行为进行约束等。规则的合理设置直接关系到系统的可用性和公平性。

基于以上分析，系统的核心应用场景可以概括为：学生通过 Web 界面查看座位的实时占用情况，选择空闲座位和合适的时间段进行预约，预约成功后按时到馆签到并使用座位；管理员在后台管理系统中查看全局数据、管理用户和座位、查看统计报表。

### 3.2 功能需求分析

#### 3.2.1 普通用户功能需求

普通用户的功能需求围绕"注册—登录—选座—预约—签到—查看—取消"这一主线展开。

用户注册与登录。用户可以通过注册功能创建账户，注册时需提供姓名、电子邮件和密码。系统使用 BCrypt 算法对密码进行加密存储。登录后，后端返回一个令牌（token），前端将其保存在 localStorage 或 sessionStorage 中，供后续请求使用。

座位查看。用户登录后可以查看座位的分布情况。系统需要按建筑、楼层、日期等维度展示座位，每个座位需明确标识其位置编号、所属区域、靠近窗户与否、是否有插座以及是否处于安静区等属性。

预约座位。用户选择一个空闲座位后，需要指定预约的日期和起止时间。系统在提交预约请求时需要进行冲突检测，确保同一座位在同一时间段内没有被其他人预约。预约成功后，座位状态更新为已预约。

个人预约记录。用户可以查看自己的历史预约记录，包括预约日期、时间、座位编号和当前状态。对于状态为"已预约"且尚未开始的记录，用户可以执行取消操作。

签到。用户在预约时段内到达图书馆后，可以通过签到功能确认使用。签到验证的逻辑是检查当前时间是否在预约的起止时间范围内。

#### 3.2.2 管理员功能需求

管理员角色负责系统的日常运营和管理，主要功能需求包括以下几个方面。

全局概览。管理员首页展示系统的关键统计数据，包括总座位数、当前预约数、今日签到数、总用户数等指标。

用户管理。管理员可以查看所有注册用户的信息，包括用户的基本资料和预约统计数据，支持对用户信息的编辑和删除操作。

座位管理。管理员可以查看所有座位的状态分布，支持对座位属性的编辑。系统中内置了座位初始化功能，用于在部署时批量创建座位数据。

预约管理。管理员可以查看当前有效的预约记录，了解各座位的预约情况，必要时可以对异常预约进行干预。

统计查看。系统提供按学生的预约统计功能，包括预约总次数、完成次数、取消次数和总预约时长等指标。管理员可以查看预约排行榜，了解座位使用热度。

#### 3.2.3 实时状态同步需求

座位预约系统的一个重要非功能性需求是状态的实时同步。当某用户预约或取消一个座位时，其他正在查看同区域座位的用户应当能够及时看到状态的变化。这一需求决定了系统不能仅依赖传统的 HTTP 轮询方式，而需要引入实时通信机制。

本系统通过 WebSocket 实现状态变化的主动推送。当发生预约、取消、签到等业务操作导致座位状态变更时，后端自动向所有已连接的 WebSocket 客户端广播更新后的座位信息。

### 3.3 非功能需求分析

系统性能方面，前端页面应在 2 秒内完成初始加载，接口响应时间控制在 500 毫秒以内，WebSocket 消息的推送延迟不超过 1 秒。

系统安全方面，用户密码需加密存储，令牌机制保证接口访问的合法性，同时防止 SQL 注入等常见 Web 攻击手段。

系统可用性方面，界面应采用清晰的信息层次和统一的设计风格，预约流程的操作路径应当简洁直接，错误信息和状态反馈应当及时明确。

系统可维护性方面，前后端代码应保持良好的模块化程度，关键业务逻辑有清晰的注释说明，数据库设计遵循规范化原则。

## 4 系统总体设计

### 4.1 系统总体架构

本系统采用前后端分离的三层架构，自下而上依次为数据持久层、后端业务层和前端展示层。

数据持久层基于 MySQL 数据库和 Spring Data JPA 技术，负责业务数据的存储和检索。该层包含 User、Seat、SeatBooking、StudentStats 和 SystemConfig 五个实体类，通过 JPA Repository 提供数据访问接口。

后端业务层基于 Spring Boot 3.2.0 构建，以 RESTful 风格提供业务接口。业务层分为控制器（Controller）、服务（Service）和数据访问（Repository）三个子层。控制器负责请求路由和参数校验，服务层封装核心业务逻辑，Repository 层提供数据访问能力。此外，后端还集成了 WebSocket 处理器，用于实时状态广播。

前端展示层基于 Vue 3 构建，使用 Vite 作为构建工具。前端采用单页面应用（SPA）模式，通过 Vue Router 管理页面路由，通过 Pinia 管理全局状态，通过 Axios 发起 HTTP 请求。

系统总体架构中，前端与后端之间的通信主要采用 HTTP RESTful 接口和 WebSocket 两种方式。HTTP 接口用于常规的业务数据操作，WebSocket 连接用于座位状态的实时推送。后端在数据库中完成数据更新后，通过 WebSocket 将变更推送到所有已连接的客户端。

这种分层架构的设计具有两个直接优势。第一，界面层与业务层相互解耦，前端页面调整不会直接影响后端服务逻辑，后端规则调整也可以通过接口形式对前端透明。第二，系统能够较容易地扩展新的访问终端，例如移动端、小程序端或自助终端，而不必重写底层业务逻辑。

### 4.2 功能模块设计

根据需求分析的结果，系统划分为以下功能模块。

用户认证模块。负责用户的注册、登录、令牌管理和密码修改功能。该模块向后端发送认证请求，后端验证通过后返回令牌，前端持令牌访问受保护的资源。本系统采用简化的令牌方案，令牌格式为"user-{id}-{UUID}"，用于在无状态的 HTTP 请求中标识用户身份。

座位展示模块。负责以网格或列表形式展示座位布局，支持按日期、时间段、建筑和楼层进行筛选。座位根据当前状态以不同颜色标识，分为"空闲""已预约""使用中"等状态。前端为每个座位渲染属性标签，如"靠窗""有插座""安静区"等，帮助用户做出选择。

预约控制模块。负责处理预约请求的核心逻辑，包括时间有效性验证、冲突检测、预约记录创建和座位状态更新。冲突检测是该模块的关键功能，通过查询指定日期和座位范围内是否存在时间区间重叠的有效预约来判断是否可以预约。

签到验证模块。负责处理用户的签到请求，验证当前时间是否在预约的起止时间范围内，签到成功后更新预约记录和座位状态。

WebSocket 通信模块。负责管理与前端的 WebSocket 连接，在座位状态发生变更时向所有连接的客户端广播更新消息。该模块使用 ConcurrentHashMap 存储会话，支持连接的建立、消息发送和断开清理。

管理员后台模块。负责用户管理、座位管理、预约管理和系统配置管理，提供统计数据查看功能。

学生统计模块。负责按用户维度聚合预约数据，统计预约总次数、完成次数、取消次数和总时长，支持排行榜查询。

### 4.3 数据库设计

本系统包含五个数据实体：用户（User）、座位（Seat）、预约记录（SeatBooking）、学生统计（StudentStats）和系统配置（SystemConfig）。

#### 4.3.1 用户表（users）

用户表存储系统注册用户的信息。主要字段包括：id（主键）、name（用户姓名）、email（登录邮箱，唯一）、password（BCrypt加密后的密码哈希值）、role（角色，取值为 USER 或 ADMIN）。此外包含 totalBookings 和 validBookings 字段用于快速查询用户的预约统计概览，createdAt 和 updatedAt 记录创建和更新时间。

角色字段区分普通用户和管理员，前端根据角色值决定可访问的路由和可执行的操作。管理员账号在系统启动时通过 AdminUserInitializer 组件自动初始化，默认邮箱为 admin@example.com。

#### 4.3.2 座位表（seats）

座位表是系统的核心资源表，存储每个座位的静态属性和动态状态。

座位编号采用"建筑-楼层-行-列"的四段式编码方案，格式如"B1-F1-A-1"，含义为 B1 栋建筑、F1 层、A 排、1 号座位。这种编码方式使得座位编号具有自解释性，通过编号即可确定座位的物理位置。

建筑编码（buildingCode）和楼层号（floorNo）支持多建筑多楼层的场景扩展。行（row）和列（col）字段定义了座位在楼层平面图中的网格坐标。

座位的静态属性包括：
- nearWindow（是否靠窗）：列号为 5 的座位被视为靠窗座位
- hasOutlet（是否有插座）：列号为奇数的座位配置有插座
- quietZone（是否属于安静区）：A 排和 B 排被划为安静区域

座位动态属性包括 status（当前状态，取值为 AVAILABLE、BOOKED、USED 等）、lastBookedBy（最后预约用户 ID）和 lastBookedAt（最后预约时间）。

座位数量方面，系统按 3 栋建筑 × 3 层楼 × 5 排（A-E）× 5 列（1-5）的配置进行初始化，共生成 225 个座位。

#### 4.3.3 预约记录表（seat_bookings）

预约记录表存储所有的预约操作记录。主要字段包括：id（主键）、seatId（座位编号，关联 seats 表）、userId（用户 ID，关联 users 表）、bookingDate（预约日期）、timeSlotId（时间段标识）、startTime（开始时间）、endTime（结束时间）、status（预约状态，取值为 BOOKED、CANCELLED、EXPIRED、CHECKED_IN 等）、checkedIn（是否已签到）。

预约记录表是系统中数据量增长最快的表，也是冲突检测的核心查询目标。每次预约请求都需要在该表中查询是否存在与目标座位、日期和时间段重叠的有效记录。

#### 4.3.4 学生统计表（student_stats）

学生统计表按用户维度聚合预约数据，用于后台管理的统计展示。主要字段包括：studentId（关联用户 ID）、studentName（用户姓名）、totalBookings（预约总次数）、completedBookings（完成次数）、cancelledBookings（取消次数）、totalHours（总预约时长）、lastBookingDate（最近预约日期）。

该表的数据通过业务操作触发更新。用户每次完成预约或取消预约时，系统同步更新对应的统计记录。这种设计避免了每次查询统计时都要扫描大量预约记录的性能问题。

#### 4.3.5 系统配置表（system_config）

系统配置表以键值对的形式存储系统运行时配置。主要字段包括：id（主键）、configKey（配置键名）、configValue（配置值）。系统中使用该表存储诚信指数阈值（integrity_threshold），默认值为 0.6，表示用户诚信指数低于 60% 时无法进行预约。

### 4.4 业务流程设计

系统的核心业务流程为座位预约流程。一个完整的预约流程包括以下步骤。

第一步，用户在前端选择目标日期，系统加载该日的座位状态数据。前端向 GET /api/seats 接口发送请求，携带 date 参数，后端返回该日期下所有座位的状态列表，每个座位标识当前是否已被预约。

第二步，用户选择空闲座位并指定预约的起止时间。前端校验时间参数（开始时间不能晚于结束时间，预约时长不超过限制），然后将预约请求发送至 POST /api/seats/{seatId}/book 接口。

第三步，后端接收预约请求后执行冲突检测。系统查询 seat_bookings 表中是否存在与目标座位、相同日期、状态为 BOOKED 且时间区间重叠的记录。重叠判定逻辑为：已存在预约的开始时间小于新预约的结束时间，且已存在预约的结束时间大于新预约的开始时间。若检测到重叠，则预约失败并返回提示信息。

第四步，冲突检测通过后，系统创建预约记录并更新座位状态。在同一个事务中完成预约记录的持久化、seat 表的状态更新和 student_stats 表的计数更新。

第五步，后端通过 WebSocket 向所有已连接的客户端广播座位状态更新消息，消息内容包括变更座位的编号、新状态和预约用户信息。

第六步，前端 WebSocket 客户端接收到广播消息后，更新本地座位数据，实现界面状态的实时刷新。

取消预约的流程相对简单。用户提交取消请求（POST /api/seats/{seatId}/cancel），后端将预约记录的状态更新为 CANCELLED，同时将座位状态恢复为 AVAILABLE，最后广播状态更新。

### 4.5 接口设计分析

系统后端以 RESTful 风格提供接口，接口路径遵循资源导向的命名规范。核心接口列表如下。

认证接口组（/api/auth）：包括 POST /api/auth/register（注册）、POST /api/auth/login（登录）、POST /api/auth/logout（注销）、GET /api/auth/me（获取当前用户信息）、POST /api/auth/change-password（修改密码）。

座位接口组（/api/seats）：包括 GET /api/seats（获取座位列表，支持日期、时间段、建筑、楼层等参数）、GET /api/seats/statistics（获取座位统计）、POST /api/seats/{seatId}/book（预约座位）、POST /api/seats/{seatId}/cancel（取消预约）、POST /api/seats/checkin/{bookingId}（签到）、POST /api/seats/initialize（初始化座位）、GET /api/seats/user/{userId}/bookings（获取用户预约记录）。

用户接口组（/api/users）：包括 GET /api/users/{id}（获取用户信息）、PUT /api/users/{id}（更新用户信息）、DELETE /api/users/{id}（删除用户）、GET /api/users/{id}/integrity（获取用户诚信指数）。

接口的请求和响应统一使用 JSON 格式。认证接口需要验证成功后返回令牌，后续受保护接口通过在 HTTP 头中携带 Authorization: Bearer {token} 进行身份验证。

## 5 系统详细设计与实现

### 5.1 用户认证模块实现

用户认证模块的核心职责是处理用户的注册、登录、令牌管理和身份验证。

注册功能接收用户提交的姓名、邮箱和密码，密码经过 BCrypt 加密后存入数据库。BCrypt 是一种自适应哈希算法，通过引入随机盐值和可配置的计算轮数来抵抗彩虹表攻击和暴力破解。在 Spring Boot 中，通过 SecurityConfig 配置类声明 BCryptPasswordEncoder 的 Bean，在 AuthServiceImpl 的 register 方法中调用 passwordEncoder.encode(password) 实现加密。

登录功能验证用户提交的邮箱和密码，验证通过后生成令牌返回给前端。令牌的生成逻辑在 generateToken 方法中实现，格式为"user-{id}-{UUID}"，其中 id 是用户的数据库主键，UUID 是随机生成的全局唯一标识符。该令牌虽然不是标准 JWT，但在毕业设计原型中能够满足基本的身份标识需求。

令牌验证通过 AuthService 的 getCurrentUserByToken 方法实现。该方法从 Authorization 请求头中提取 Bearer 前缀后的令牌字符串，解析出用户 ID，在内存中查找对应的会话信息。如果令牌有效，返回用户信息；否则返回 401 状态码。

前端通过 Pinia 的 auth store 管理认证状态。登录成功后，令牌被保存到 localStorage（选择"记住我"时）或 sessionStorage 中。Axios 请求拦截器在每个 HTTP 请求的头部自动添加 Authorization 字段。Vue Router 的 beforeEach 守卫检查路由的 meta.requiresAuth 属性，对于需要认证的页面，如果本地没有令牌则重定向到登录页。

从实现角度看，该认证方案具有一定的教学与演示价值。一方面，系统已经实现了密码加密存储，避免明文泄露；另一方面，令牌结构保持轻量，有助于前后端快速联调。

### 5.2 座位预约模块实现

座位预约模块是系统的核心业务模块，其关键实现在 SeatService 类的 bookSeatWithDuration 方法中。

该方法接收五个参数：seatId（座位编号）、userId（用户 ID）、bookingDate（预约日期）、startTime（开始时间）、endTime（结束时间）。方法内部按以下步骤执行：

第一步，参数校验。检查所有参数是否为 null，以及开始时间是否早于结束时间、预约时段是否在系统允许的时间范围内。若校验不通过，方法返回 false。

第二步，座位查找。通过 seatRepository.findBySeatId(seatId) 从数据库获取座位实体，若座位不存在则返回 false。

第三步，冲突检测。这是预约模块最为关键的一步。系统调用 seatBookingRepository 的 existsBySeatIdAndBookingDateAndStatusAndStartTimeLessThanAndEndTimeGreaterThan 方法，查询是否存在与目标座位、相同日期、状态为 BOOKED、且时间区间重叠的预约记录。具体来说，该方法的查询条件是：已存在预约的开始时间小于新预约的结束时间，且已存在预约的结束时间大于新预约的开始时间。当两个条件同时满足时，说明时间区间存在重叠，当前预约请求被拒绝。

第四步，预约记录创建。冲突检测通过后，创建一个新的 SeatBooking 实体，设置 seatId、userId、bookingDate、startTime、endTime、timeSlotId 和 status（BOOKED）等字段，然后调用 seatBookingRepository.save(booking) 持久化到数据库。

第五步，座位状态更新。更新座位实体的 status 字段为 BOOKED，记录 lastBookedBy 和 lastBookedAt 时间，然后保存座位实体。

第六步，实时通知。调用 SeatStatusWebSocketHandler.broadcastSeatStatusUpdate(seat) 方法广播状态变更。

当同一座位在同一日期存在时间重叠的预约时，冲突检测会准确拦截。

取消预约的实现在 cancelBooking 方法中。方法接收 bookingId 作为参数，查询对应的预约记录，将状态更新为 CANCELLED，同时将对应座位的状态恢复为 AVAILABLE，同样触发 WebSocket 广播。

### 5.3 诚信指数与签到实现

诚信指数是系统用于约束用户预约行为的一种评价机制，定义在 getUserIntegrity 方法中。该方法的计算逻辑是：从用户信息中获取 totalBookings 和 validBookings，诚信指数 = validBookings / totalBookings。当 totalBookings 为 0 时，诚信指数默认为 1.0。

在预约接口中，系统在冲突检测之前会先检查用户的诚信指数。从 SystemConfig 表中读取 integrity_threshold 配置值（默认为 0.6），若用户诚信指数低于阈值，预约请求被拒绝。这一机制可以有效约束频繁取消预约的行为，鼓励用户遵守预约规则。

签到功能实现在 checkin 方法中。该方法接收 bookingId 和 userId 两个参数，首先查询预约记录是否存在且状态为 BOOKED，然后验证当前时间是否在预约的起止时间范围内。验证通过后将预约记录的 checkedIn 字段设为 true，状态更新为 CHECKED_IN。

### 5.4 WebSocket 实时状态同步实现

WebSocket 通信模块是整个系统中实现实时状态同步的关键组件。该模块由 SeatStatusWebSocketHandler 和 WebSocketConfig 两个类组成。

WebSocketConfig 类负责注册 WebSocket 端点，将 SeatStatusWebSocketHandler 映射到 /ws/seat-status 路径，并设置允许跨域访问的来源。

SeatStatusWebSocketHandler 类继承自 TextWebSocketHandler，核心功能包括会话管理和消息广播两个部分。

在会话管理方面，处理器使用 ConcurrentHashMap<String, WebSocketSession> 存储所有已连接的 WebSocket 会话。当客户端建立连接时（afterConnectionEstablished），将客户端的 session 对象加入 Map 中；当连接断开时（afterConnectionClosed），从 Map 中移除对应的 session。ConcurrentHashMap 的选择保证了在多线程环境下的线程安全。

在消息广播方面，broadcastSeatStatusUpdate 方法是核心。当座位状态发生变更时（预约、取消等操作），系统调用该方法，构建一个包含座位信息的消息对象（类型为 seatStatusUpdate，包含 seatId、status、buildingCode、floorNo 等字段），然后将消息以 JSON 格式发送给 Map 中所有有效的 session。发送过程中捕获 IOException 异常，对于连接已关闭的客户端 session，将其从 Map 中移除。

前端通过 WebSocket 管理器（websocket.js）与后端建立连接。该管理器采用类封装的方式，提供 connect、disconnect、onMessage、sendMessage 等接口方法。为了使组件能够独立处理消息，管理器设计了基于事件类型的监听器机制，支持注册回调函数。在连接管理方面，管理器实现了指数退避重连策略：当连接意外断开时，等待时间从 3 秒开始逐次递增，最多尝试 5 次重连。同时，管理器通过手动关闭标记区分主动关闭和意外断开，避免在主动注销时进行不必要的重连。

当 WebSocket 客户端收到 seatStatusUpdate 消息后，解析消息内容，根据座位编号在本地座位数据中查找并更新对应座位的状态，触发 Vue 的响应式更新机制，界面自动反映状态变化。

### 5.5 管理员后台模块实现

管理员后台模块由用户管理、座位管理、预约管理和统计管理四个子模块组成。

用户管理基于 UserController 提供接口，管理员可以查看用户列表、编辑用户信息、删除用户。用户信息中包含了 totalBookings 和 validBookings 字段，方便管理员了解用户的预约活跃度。在删除用户时，需要确保该用户没有未完成的预约。

座位管理基于 SeatController 提供接口，管理员可以查看所有座位的状态分布、编辑座位属性。座位初始化功能（POST /api/seats/initialize）在部署时调用，按照预设的建筑、楼层、行列配置批量生成座位数据。随机化属性功能（POST /api/seats/randomize-attributes）用于测试阶段，为座位随机分配靠近窗户、有插座和安静区等属性。

预约管理提供按状态筛选的预约记录查询功能（GET /api/seats/bookings?status=BOOKED），管理员可以查看当前有效的预约信息，包括预约人、座位编号、预约时间段等。

#### 5.5.1 统计数据生成机制

学生统计模块围绕用户维度聚合预约数据，统计内容包括预约总数、完成数、取消数、总时长和最近预约日期。系统在统计表为空时，支持从预约记录表重建统计数据。这种设计能够降低统计数据与原始数据脱节的风险。

统计数据通过 StudentStatsController 以 REST 接口的形式暴露，支持单用户查询、按姓名查询、排行榜查询（GET /api/student-stats/top）和系统总体统计（GET /api/student-stats/system）。排行榜功能按预约总次数降序排列，为管理员识别高频使用用户提供了数据支撑。

### 5.6 安全性与异常处理实现分析

虽然本系统的主要目标是完成图书馆座位预约业务，但系统实现中仍然体现了若干基础安全与异常处理思路。

在用户认证方面，系统使用 BCrypt 对密码进行加密，避免明文存储带来的直接安全风险。在接口调用方面，前端 Axios 拦截器为每个请求自动添加 Authorization 请求头，后端通过令牌解析完成身份识别。

在 Web 安全方面，SecurityConfig 配置了 CORS（跨域资源共享），允许来自 localhost:3000 和 localhost:3001 的请求。系统目前对所有请求采用 permitAll() 策略，即所有接口均可被访问，通过前端路由守卫和后端业务校验共同控制操作权限。

异常处理方面，系统已经能够覆盖常见的业务失败场景，例如预约时间格式错误、时间段冲突、取消目标不存在等，并通过统一的消息文本反馈给前端。此类处理虽然在形式上较为直接，但在业务系统中具有重要意义，因为它保证了用户在失败情况下仍能得到可理解的反馈。

## 6 系统测试与结果分析

### 6.1 测试环境

测试环境如下。前端构建环境为 Node.js（通过 Vite 构建工具），后端构建环境为 Maven（JDK 17），运行环境为 Spring Boot 3.2.0 内嵌 Tomcat，数据库采用 MySQL。前端构建命令为 npm run build，后端打包命令为 mvn package。

### 6.2 构建与打包验证

前端构建结果：执行 npm run build 后，Vite 成功完成项目构建。构建产物中，JavaScript 文件约 213KB（gzip 后约 77KB），CSS 文件约 44.62KB（gzip 后约 7.86KB），入口 HTML 文件约 0.47KB。构建耗时约 1.12 秒。这表明前端应用在打包后保持了较小的资源体积，有利于页面加载性能。

后端打包结果：执行 mvn -DskipTests package 后，成功生成 springboot-vue3-system-1.0.0.jar 文件，体积约 62MB。该 JAR 文件包含了 Spring Boot 应用的所有依赖和嵌入式 Tomcat 服务器，可以直接通过 java -jar 命令启动。

### 6.3 接口功能测试

测试方法是通过 curl 命令对核心接口进行请求，验证响应结果是否符合预期。测试用例覆盖了注册登录、座位查看、预约、冲突检测、取消预约和 WebSocket 状态同步等关键业务链路。

用户认证测试。调用 POST /api/auth/register 接口注册新用户，返回成功响应。调用 POST /api/auth/login 接口，传入正确的邮箱和密码，返回包含令牌和用户信息的响应。调用 GET /api/auth/me 接口并携带令牌，返回当前登录用户的信息。

座位查询测试。调用 GET /api/seats 接口查询座位列表，返回包含座位编号、状态、静态属性等字段的完整数据。调用 GET /api/seats/statistics 接口返回座位的统计数据，格式为 total、booked、available、used 四个指标。系统初始化后总座位数为 400，另有历史遗留座位 11 个。

座位预约测试。以 seatId 为 B1-F1-A-1 的座位为目标，调用 POST /api/seats/{seatId}/book 接口，传入用户 ID、日期、起止时间等参数。系统返回成功响应，座位状态更新为 BOOKED，预约记录写入数据库。

冲突检测测试。对同一座位（B1-F1-A-1）在同一日期提交时间区间重叠的预约请求（09:30-10:30，与已存在的 09:00-11:00 重叠）。系统返回 success 为 false 的响应，表明冲突检测正确拦截了重复预约。对同一日期不同座位的预约请求，系统正常处理，说明冲突检测的范围限定在同一座位内。

取消预约测试。使用返回的 bookingId 调用取消接口后，系统返回"预约已取消"。再次查询座位统计信息，已预约数量减少，空闲数量恢复，说明状态回滚正确。

### 6.4 测试结果分析

从测试结果可以看出，系统已经实现了从身份认证到预约执行再到状态回滚与广播同步的完整业务闭环。

首先，注册和登录测试说明系统具备基本的身份管理能力。BCrypt 加密方案确保了密码存储的安全性，令牌机制支持无状态的接口鉴权。

其次，预约与冲突测试表明系统能够针对同一座位的重叠时段进行准确判断，避免超量预约。冲突检测的核心是时间区间重叠判定，测试中两个重叠时段（09:00-11:00 和 09:30-10:30）被成功拦截，而不同座位之间的预约则互不影响，符合预期。

再次，取消预约后的状态回滚验证了系统的数据一致性维护能力。取消操作同时更新了预约记录状态和座位状态，没有出现状态不一致的情况。

不过，测试结果也揭示出若干进一步优化的方向。第一，当前测试主要基于接口级验证，尚未覆盖高并发场景下的竞态冲突问题。第二，后端认证仍采用简化令牌方案，接口虽然可用，但安全边界不够严格。第三，统计模块能够完成基础汇总，但在预约时长精细计算和行为模式分析方面还可以进一步完善。

### 6.5 本章小结

本章基于实际的构建与接口验证结果，对系统的主要功能进行了测试分析。测试结果表明，系统已经能够完成普通用户注册登录、座位初始化、预约冲突控制、取消预约等关键功能，说明系统具备基本的业务可用性。

## 7 结论

本文围绕"基于 Vue 的图书馆座位预约系统的研究与实现"这一课题，结合高校图书馆高频自习与考研备考场景，对图书馆座位预约系统进行了较为系统的分析、设计与实现。

全文首先从实际应用背景出发，分析了图书馆座位管理中存在的信息滞后、预约冲突频发等问题，阐明了构建信息化预约系统的必要性。在技术层面，对 Vue 3、Spring Boot、Spring Data JPA、WebSocket 和前后端分离架构等关键技术进行了介绍，明确了系统的技术选型依据。

在需求分析阶段，围绕普通用户和管理员两类角色的业务场景，对系统的功能需求和非功能需求进行了详细的梳理和界定。在系统设计阶段，从架构设计、模块划分、数据库设计、业务流程设计和接口设计五个方面展开了系统的总体设计工作。

在实现阶段，对用户认证、座位预约、诚信指数与签到验证、WebSocket 实时同步和管理员后台等核心模块的实现逻辑进行了分析。系统实现了基于时间区间重叠判定的冲突检测机制，保证了预约数据的准确性；利用 WebSocket 实现了座位状态的实时推送，提升了多用户场景下的信息同步效率。在数据库层面，系统设计了用户、座位、预约记录、学生统计和系统配置五张核心表，支撑了预约业务的数据流转。

在测试阶段，通过实际的构建验证和接口测试，对系统的核心功能进行了检验。前端构建耗时 1.12 秒，产物体积约 258KB（JS + CSS），后端 JAR 包体积约 62MB。接口测试覆盖了注册、登录、预约、冲突检测和取消预约等关键环节，验证结果符合预期。

当然，本文实现的系统仍属于面向本科毕业设计的工程原型，在安全性、规则完备性和数据分析深度等方面还有进一步提升空间。后续可以围绕标准化鉴权、高并发预约控制、签到违约约束、统计建模细化和多终端联动等方向继续扩展。总体而言，本文的研究与实现表明，基于前后端分离架构的图书馆座位预约系统在技术上可行、在业务上有实际应用价值。
