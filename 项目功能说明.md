# 项目功能说明

## 1. 项目概述
本项目是一个面向图书馆场景的座位预约管理系统，采用 `Spring Boot + Vue 3` 前后端分离架构实现，核心目标是解决传统图书馆座位管理中“座位空置难判断、预约冲突难控制、管理员统计不直观、状态更新不及时”等问题。

系统面向两类用户：

- 普通用户：完成注册登录、查询座位、按日期与时段预约座位、查看个人预约记录、取消预约。
- 管理员：查看系统运行概况、管理用户、管理座位、查看预约记录、查看学生统计数据。

本项目除了基本 CRUD 功能外，还实现了较为完整的预约时段控制、预约冲突判定、预约状态自动更新、WebSocket 实时广播、学生预约统计等功能，适合作为“基于 Vue 的图书馆座位预约系统”的本科毕业论文选题基础。

## 2. 技术架构

### 2.1 总体架构
- 前端：`Vue 3 + Vite + Vue Router + Pinia + Axios`
- 后端：`Spring Boot 3 + Spring MVC + Spring Data JPA + Spring Security`
- 数据库：`MySQL`，同时引入 `H2` 作为运行时依赖
- 实时通信：`WebSocket`
- 构建工具：前端使用 `Vite`，后端使用 `Maven`
- 开发语言：前端 `JavaScript`，后端 `Java 17`

### 2.2 分层设计
- 表现层：Vue 页面、路由守卫、状态管理、表单校验、座位布局可视化。
- 接口层：Spring Boot Controller 提供 RESTful API。
- 业务层：Service 负责认证、预约校验、状态变更、统计处理等核心逻辑。
- 数据访问层：Repository 基于 JPA 操作数据库。
- 数据层：用户表、座位表、预约记录表、学生统计表。

### 2.3 前后端协作方式
- 前端通过 Axios 调用 `/api` 接口。
- 登录后令牌保存在 `localStorage` 或 `sessionStorage` 中，并通过请求拦截器自动附带 `Authorization` 请求头。
- 路由层根据登录状态和角色信息完成页面访问控制。
- 座位状态变化后，后端通过 WebSocket 向前端广播，前端接收后局部刷新页面。

## 3. 业务角色与核心流程

### 3.1 用户角色
- `USER`：普通用户，具有预约、查看、取消个人预约等权限。
- `ADMIN`：管理员，具有用户管理、座位管理、预约管理、统计查看等权限。

### 3.2 核心业务流程

#### 3.2.1 用户预约流程
1. 用户注册或登录系统。
2. 进入座位预约页面，选择预约日期。
3. 选择座位后，系统读取该座位当日已预约的时间段。
4. 用户选择开始时间与结束时间。
5. 后端执行时间合法性校验和冲突检测。
6. 预约成功后保存记录，并更新座位状态。
7. 系统通过 WebSocket 向客户端广播最新座位状态。

#### 3.2.2 管理员管理流程
1. 管理员登录后台控制台。
2. 查看系统总用户数、今日预约数、总座位数、使用率等指标。
3. 按条件筛选预约记录，必要时取消预约。
4. 维护用户信息、座位信息和统计信息。
5. 对异常座位状态执行初始化、刷新、重置等管理操作。

## 4. 后端功能说明

### 4.1 认证与授权模块
- 用户可通过邮箱和密码进行登录。
- 用户注册时进行邮箱唯一性校验，默认角色为 `USER`。
- 密码采用 `BCryptPasswordEncoder` 加密存储。
- 提供登录、注册、登出、获取当前用户、刷新令牌、忘记密码、重置密码等接口。
- 当前项目中的令牌实现为简化版字符串令牌，格式类似 `user-{id}-{UUID}`，并非完整 JWT 鉴权方案。
- 后端已引入 Spring Security，但当前安全配置为 `permitAll`，主要权限控制放在前端路由层，适合课程设计与原型系统，不适合直接用于生产环境。

### 4.2 用户管理模块
- 支持用户信息新增、查询、修改、删除。
- 支持按邮箱查找用户，用于登录和注册校验。
- 管理员可在后台查看用户列表、用户详情、用户角色及注册时间。
- 系统启动时会自动检测管理员账户，不存在时自动创建默认管理员：
  - 邮箱：`admin@example.com`
  - 密码：`123123`

### 4.3 座位管理模块
- 支持座位初始化，默认生成 `5` 行 `8` 列共 `40` 个座位，编号格式为 `A-1` 至 `E-8`。
- 支持按主键 ID 或业务编号 `seatId` 查询座位。
- 支持按座位状态、行列条件筛选座位。
- 支持创建、更新、删除座位。
- 支持统计可用座位、已预约座位、已使用座位和总座位数。
- 座位实体包含如下关键字段：
  - `seatId`：业务编号
  - `row`、`col`：座位位置
  - `status`：座位状态
  - `lastBookedBy`、`lastBookedAt`：最近预约人及预约时间
  - `createdAt`、`updatedAt`：创建与更新时间

### 4.4 座位预约模块
- 支持按“日期 + 开始时间 + 结束时间”方式预约座位。
- 系统营业时间限制为 `08:00-22:00`。
- 时间粒度控制为 `30` 分钟。
- 最短预约时长为 `30` 分钟。
- 后端通过时间区间重叠检测判断是否与已有预约冲突。
- 同一座位同一天可存在多个不冲突的预约时段。
- 支持两种取消方式：
  - 推荐方式：按 `bookingId` 取消预约
  - 兼容方式：按 `seatId + date + startTime + endTime` 取消预约
- 支持查询：
  - 全部预约记录
  - 当前有效预约记录
  - 用户个人预约记录
  - 某座位某日预约记录

### 4.5 预约状态控制逻辑
系统中涉及两类状态：座位状态与预约记录状态。

#### 4.5.1 座位状态
- `AVAILABLE`：当前可预约
- `BOOKED`：当日存在预约
- `FUTURE_BOOKED`：未来日期存在预约
- `USED`：已使用

#### 4.5.2 预约记录状态
- `BOOKED`：已预约
- `CANCELLED`：已取消
- `EXPIRED`：已过期

#### 4.5.3 状态自动维护机制
- 查询当日座位时，系统会自动扫描已过结束时间的预约，并将其更新为 `EXPIRED`。
- 取消预约后，系统会重新计算该座位在指定日期的状态。
- 若当天已无预约但未来仍有预约，则座位状态切换为 `FUTURE_BOOKED`。
- 若既无当天预约也无未来预约，则状态恢复为 `AVAILABLE`。

### 4.6 学生统计模块
- 支持学生统计信息的增删改查。
- 支持按统计记录 ID、学生 ID、学生姓名进行查询。
- 支持按预约次数降序生成排行榜。
- 支持系统总体统计：
  - 学生总数
  - 预约总数
  - 完成预约数
  - 取消预约数
  - 平均时长
- 当统计表为空时，可根据预约记录自动重建统计数据。
- 当前实现中，统计逻辑采用简化处理：
  - `BOOKED` 被计为完成预约
  - 总时长在部分逻辑中按固定值估算
  - 适合论文中作为“当前实现与后续优化空间”的分析点

### 4.7 WebSocket 实时通知模块
- 后端注册 WebSocket 通道 `/ws/seat-status`。
- 座位状态发生变化时，系统向所有在线会话广播 `seatStatusUpdate` 消息。
- 前端接收到消息后局部更新座位状态，减少整页刷新。
- WebSocket 管理器具备断线重连能力，最多重试 `5` 次。

### 4.8 配置与部署相关说明
- 数据源默认指向本地 MySQL 数据库 `test`。
- JPA 配置为 `ddl-auto=update`，可自动维护表结构。
- 后端配置文件中服务端口为 `8081`。
- 前端 WebSocket 地址当前写为 `ws://localhost:8080/ws/seat-status`，与后端端口配置存在不一致，部署时需要统一。
- 该不一致现象可作为论文中“系统测试与问题分析”部分的一个实现细节。

## 5. 前端功能说明

### 5.1 认证与会话管理
- 登录页支持邮箱密码登录、记住我、前端表单校验。
- 注册页支持用户名、邮箱、密码、确认密码、协议勾选等校验。
- 登录成功后根据用户角色跳转到普通用户页面或管理员控制台。
- 支持从本地存储恢复登录态。
- 当接口返回 `401` 时，前端会自动清理登录信息并跳转到登录页。

### 5.2 路由与权限控制
- 路由页面包括：首页、登录页、注册页、座位预约页、管理员页、用户列表页、用户详情页。
- 普通用户访问受保护页面时，若未登录将被重定向到登录页。
- 管理员页面要求 `ADMIN` 角色。
- 已登录用户再次访问登录页或注册页时，会被自动跳转到对应主页。

### 5.3 座位预约页面
- 采用座位网格展示方式，按 `A-E` 行与 `1-8` 列呈现座位布局。
- 支持选择日期后查看该日座位可预约情况。
- 支持查看某个座位的当日预约时间段列表。
- 支持根据已存在预约动态计算可选开始时间与结束时间。
- 提供个人预约记录、账户信息、座位时间轴可视化等功能。
- 支持自动轮询刷新和 WebSocket 实时刷新相结合的方式保持页面状态更新。
- 采用不同样式区分：
  - 可预约
  - 部分占用
  - 本人已预约
  - 已满

### 5.4 管理员控制台
- 采用单页控制台形式，包含多个业务标签页。
- 仪表盘展示系统总览与近期活动。
- 预约管理支持按日期、状态、关键词筛选预约记录。
- 用户管理支持新增、编辑、删除、查看用户信息。
- 座位管理支持初始化座位、刷新座位、重置座位状态、查看最近预约信息。
- 学生统计页面支持统计列表与排行榜展示。
- 系统设置页面支持系统名称、最大预约数、预约时间范围、通知开关等配置项展示。
- 当前“系统设置”保存在浏览器 `localStorage` 中，属于前端演示型配置，尚未持久化到后端。

## 9. 可支撑论文写作的重点分析点
如果将本项目用于本科毕业论文，建议重点突出以下技术问题，而不是仅停留在功能罗列层面：

- 前后端分离架构下的图书馆座位预约系统设计。
- 基于时间区间重叠检测的预约冲突控制机制。
- 面向多时段预约的座位状态建模与动态刷新策略。
- 基于 WebSocket 的座位状态实时同步实现。
- 面向管理员决策的学生预约统计与可视化管理。

## 10. 当前实现的局限与可优化方向
- 当前令牌机制为简化实现，可进一步升级为标准 JWT + 过滤器鉴权。
- 后端接口权限控制尚未做细粒度限制，主要依赖前端路由守卫。
- 学生统计模块中部分时长计算逻辑较为粗略，可进一步按真实预约时长累计。
- 系统设置仅保存在前端本地，尚未形成后端持久化配置中心。
- WebSocket 地址与后端端口配置存在不一致，部署时需要统一处理。
- 目前尚未引入更完整的预约违约、签到、超时释放、黑名单等图书馆业务规则。
